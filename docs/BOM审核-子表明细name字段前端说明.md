# BOM 审核接口：子表明细 `name` 字段 — 前端说明

面向接口：`POST /api/method/bairun_erp.utils.api.material.bom_item_list.audit_so_bom_list`  
本文说明 **`details[]` 每一行为何要尽量带上 `name`（或与 `name` 等价的标识）**，避免重复行、错行更新等问题。

---

## 1. `name` 是什么？

- **`name` 是 Frappe 子表文档 `BR SO BOM List Details` 在数据库里的主键**（每一行明细一条记录对应一个唯一 `name`）。
- 该值由系统在首次插入明细时自动生成（一般为无业务含义的字符串），**在同一条 BOM 清单生命周期内应视为“行的稳定 ID”**。
- 业务含义上：**它标识的是「这一条明细行」本身**，而不是物料编码、也不是行号展示顺序。

---

## 2. 两种明细行（前端约定：有 `name` / 无 `name`）

业务上 **`name` 只分两种用法**，请与后端 Upsert 行为对齐：

| 情况 | `name`（或 `id`） | 含义 |
|------|-------------------|------|
| **① 已有行** | **有**（非空） | 表示该行**早已存在于** `BR SO BOM List Details`，本次审核只做**字段更新**，**禁止当成新行**。 |
| **② 新增行** | **没有**（不传、或空） | 表示这是用户在页面上**手工新增**的明细，库里尚无对应子表记录，后端才应按**新增**处理。 |

**约定总结：**

- **从接口/库中加载出来的旧行**：必须带上 **`name` 或 `id`** → **只更新**。
- **只有「没有 `name`」这一种情况**，才表示你要**新增**一行；不要对已有行省略 `name`，否则后端可能用业务键去猜，猜错就会多插一行。

---

## 3. 审核提交时为什么要对「已有行」必传 `name`？

后端实际逻辑仍是 **Upsert**，顺序为：

1. **有 `name`（或 `id`）且能命中** → **只更新这一行**（与上表 ① 一致）。
2. **没有 `name`** → 再用业务键 **`(row_no, item_code, bom_code, level)`** 尝试命中更新。
3. **仍匹配不上** → **`append` 新行**（与上表 ② 的新增同源，但也可能是**误匹配失败导致的重复插入**）。

因此：**已有行若漏传 `name`**，在合并表格、重排行号或包材块漏 `bom_code`/`level` 时，极易 **误判为新增 → 重复明细**（例如同一包材两行）。

---

## 4. 和查询接口的对应关系

拉取 BOM 明细时（例如 `get_product_bom_list_new`），子表行的 `name` 在 JSON 里往往以字段 **`id`** 返回（与 Desk 里子表行的 `name` 为同一值）。

| 场景 | 建议 |
|------|------|
| 列表/详情里用的是 `id` | 审核提交时在 `details[]` 中传 **`name`**，值为该行的 **`id`**（二者同一串即可）。 |
| 列表/详情里已有 `name` | 直接原样回传 **`name`**。 |

后端在解析明细时，**同时接受 `name` 与 `id`**（任填其一即可作为子表行标识，见实现 `_normalize_detail_row`）。

---

## 5. 请求示例（推荐写法）

```json
{
  "json_data": {
    "sales_order_no": "SAL-ORD-2026-00001",
    "item_code": "成品编码",
    "header": { "grossMargin": 12.5 },
    "details": [
      {
        "name": "abc12xyz34",
        "rowNo": 6,
        "itemCode": "BLISTER-3-3-3",
        "bomCode": "A1-P1",
        "itemGroup": "包材",
        "ratioQty": 2000
      }
    ],
    "mark_approved": 1
  }
}
```

或使用查询返回的 `id`：

```json
{
  "details": [
    {
      "id": "abc12xyz34",
      "itemCode": "BLISTER-3-3-3"
    }
  ]
}
```

（后端会把 `id` 当作子表 `name` 使用。）

**新增一行（手工新增）示例** — 不传 `name` / `id`，由后端插入：

```json
{
  "details": [
    {
      "itemCode": "NEW-RM-001",
      "level": 2,
      "bomCode": "A1-9",
      "itemName": "新拆分的料",
      "ratioQty": 0.5
    }
  ]
}
```

---

## 6. 与「一键生单」回写的关系

一键生单回写采购单号时，**最稳妥的关联方式**也是子表行的 `name`（见《BOM一键生单-后端改造完成说明》中的 `br_so_bom_list_detail_name`）。  
审核阶段若始终保持 **`name` 不丢**，后续生单与列表展示更容易与库内唯一一行对齐。
